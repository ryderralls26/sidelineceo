Root Cause of Login Failures\n\n**CRITICAL ISSUE IDENTIFIED:**\n\nThe application had **TWO COMPETING AUTHENTICATION SYSTEMS**:\n\n1. **Legacy Client-Side System** (`lib/storage.ts`)\n   - Stored users in browser `localStorage` with **plaintext passwords**\n   - Key: `\"users\"`\n   - Security Risk: Passwords visible in browser dev tools\n\n2. **Server-Side System** (`app/api/auth/*`)\n   - Uses **PostgreSQL/SQLite** database via Prisma ORM\n   - Passwords hashed with **bcrypt (10 rounds)**\n   - Sessions stored in **httpOnly secure cookies** (not accessible to JavaScript)\n\n**Failure Scenario:**\n1. User credentials created via old localStorage system\n2. Login form calls `AuthContext.login()`\n3. `AuthContext.login()` calls `/api/auth/signin` server endpoint\n4. Server queries **Prisma database** (empty or has different users)\n5. **Database lookup fails** -> 401 Unauthorized\n6. localStorage has credentials but server ignores it\n\n**Root Cause:** The server-side auth routes query the database, but users were only created in localStorage. The systems were **not synchronized**.\n\n---\n\n### 4. Current Source of Truth\n\n**BEFORE FIXES:**\n- **Client-side (INSECURE):** `localStorage` via `lib/storage.ts` - plaintext passwords ❌\n- **Server-side (SECURE):** PostgreSQL/SQLite via Prisma - bcrypt hashed passwords ✅\n- **No database connection configured** (missing .env)\n\n**AFTER FIXES:**\n- **Server-side ONLY:** SQLite database (`dev.db`) via Prisma\n- **httpOnly cookies** for session management\n- **NO client-side password storage**\n\n---\n\n### 5.
